# 商品销售分析报告模板

## 报告标题

标题应直接回答“分析对象 + 时间范围 + 核心问题”。

示例：

- 近 30 天商品销售变化原因分析
- 某类目促销期间销售与毛利表现分析
- 某渠道商品销售下滑原因分析
- 某批商品库存积压与销售表现分析

标题要求：

- 不写泛泛的“数据分析报告”。
- 不写与用户问题无关的范围。
- 如果用户问题很明确，标题必须直接包含该问题的核心对象，例如“销售下滑”“促销效果”“库存积压”“复购下降”。

---

## 执行摘要

篇幅建议：150~250 字。

必须包含：

- 本次回答的业务问题。
- 最关键的 2~4 条结论。
- 每条核心结论的结论强度：强 / 中 / 弱。
- 一句限制说明。

写法要求：

- 先结论，后证据。
- 不写背景铺垫，不写“随着市场竞争加剧”等空话。
- 不引入正文没有的结论。
- 没有证据支撑的内容不写。
- 摘要必须优先回答用户最关心的问题，而不是先复述一堆通用指标。

推荐写法：

```md
本次分析关注【用户问题】。从现有数据看，最主要的发现是：【结论 1】（强/中/弱），主要由【关键数据】支持；其次，【结论 2】（强/中/弱），表现为【关键数据】。需要注意的是，【限制说明】，因此关于【受影响结论】仍需谨慎解释。
```

---

## 问题与范围

建议篇幅：80~150 字。

必须写清：

- 用户原始问题。
- 本次实际回答的问题。
- 时间范围。
- 数据来源。
- 业务对象范围，例如商品、类目、渠道、门店、地区、客群。
- 关键统计口径，例如是否包含退款、是否包含税费和运费、订单纳入口径。

写法要求：

- 不要写过长背景。
- 不要把系统做了什么流程写成报告内容。
- 如果用户问题很窄，范围也要收窄，不要强行扩展成全局分析。
- 如果数据不足以完整回答用户问题，要在这里说明“本报告只能回答其中的哪一部分”。

---

## 核心发现

核心发现是报告主体，必须围绕用户问题展开。

如果用户问“为什么销售下降”，核心发现就要围绕销售下降的原因展开。  
如果用户问“哪个商品卖得最好”，核心发现就要围绕商品排名、贡献占比、销售质量展开。  
如果用户问“促销有没有效果”，核心发现就要围绕促销前后、促销与非促销对比、销售和毛利变化展开。  
如果用户问“库存是否有问题”，核心发现就要围绕库存、销量、周转、缺货或积压展开。

禁止行为：

- 不要先写大量与用户问题无关的通用经营分析。
- 不要在用户关心的问题上只写一两句话。
- 不要只在“关键证据”表格中给数据，正文必须结合数据解释。
- 不要把所有指标平均分配篇幅，应该把最多篇幅给用户最关心的问题。
- 不要为了模板完整而强行写无关模块。

### 核心发现写作结构

核心发现建议包含 2~4 个小节，每个小节对应一个主要发现。

每个小节必须包含：

1. 一个明确的小结论。
2. 与该结论直接相关的数据。
3. 一个局部数据表格。
4. 对表格的解释。
5. 结论强度或限制说明。

推荐结构：

```md
### 发现一：【直接回答用户问题的结论】

先用 2~4 句话说明这个发现，必须包含关键数字。

| 维度/对象 | 指标 1 | 指标 2 | 对比值 | 变化 | 说明 |
| -------- | -----: | -----: | -----: | ---: | ---- |
|          |        |        |        |      |      |

解释：
- 这组数据说明……
- 对用户问题的含义是……
- 结论强度为【强/中/弱】，原因是……
```

## 图表引用要求

如果前序图表生成节点已经生成图表，请根据报告内容按需引用。引用图表时必须使用 Markdown 图片语法：

```
![图表标题](图片地址)
```

图表应放在与其结论最相关的位置，而不是集中堆放在报告末尾。每张图表下方需要补充 1-3 句话说明，解释图表展示了什么、它支持了哪个分析结论，以及是否存在数据口径或样本限制。

不要引用不存在的图片地址；不要编造图表内容。如果某张图表与当前报告结论关系不强，可以不引用。

### 局部数据表格要求

核心发现部分必须根据用户问题插入 1~3 个局部数据表格。

局部数据表格不同于“关键证据”总表，它应该服务于当前小节的解释。

表格选择规则：

- 如果用户问销售变化：展示当前周期 vs 对比周期的销售净额、订单数、售出件数、客单价、折扣率、退款率。
- 如果用户问商品表现：展示 Top 商品 / Bottom 商品的销售净额、销量、贡献占比、毛利率、退款率。
- 如果用户问类目表现：展示各类目的销售净额、贡献占比、增长率、毛利率。
- 如果用户问渠道 / 门店 / 地区：展示各渠道 / 门店 / 地区的销售额、订单数、客单价、贡献占比、变化率。
- 如果用户问促销效果：展示促销组 vs 非促销组，或促销前 vs 促销后的销售额、订单数、折扣率、毛利率、退款率。
- 如果用户问库存：展示商品或类目的库存件数、平均日销量、库存可支撑天数、售罄率、缺货天数。
- 如果用户问客户：展示新客、老客、复购率、人均订单数、客户平均消费额。
- 如果用户问广告：展示广告展示量、点击量、点击率、花费、转化数、CPA、ROAS。

表格规模控制：

- 默认每个局部表格展示 5~10 行。
- 如果数据很多，只展示最相关的 Top N / Bottom N / 异常项。
- 如果完整结果已经保存为文件，应在表格后说明：“完整明细可在结果文件中查看。”
- 不要在正文中展示超大表格。
- 不要为了展示表格而展示无关数据。

### 整体经营结果

适用场景：

- 用户询问整体销售表现。
- 用户询问销售上涨或下降。
- 用户询问经营结果是否变好。
- 用户没有明确指定某个商品、类目或渠道，需要先看整体。

重点写：

- 销售净额（net_sales）、订单数（orders）、售出件数（units_sold）、净额口径客单价（aov_net）的核心变化。
- 是否同时影响毛利 / 利润。
- 当前变化主要是规模变化、价格变化还是结构变化。
- 这些变化与用户问题之间的关系。

推荐表格：

| 指标                              | 当前值 | 对比值 | 变化量 | 变化率 | 解释 |
| --------------------------------- | -----: | -----: | -----: | -----: | ---- |
| 销售净额（net_sales）             |        |        |        |        |      |
| 订单数（orders）                  |        |        |        |        |      |
| 售出件数（units_sold）            |        |        |        |        |      |
| 净额口径客单价（aov_net）         |        |        |        |        |      |
| 折扣率（discount_rate）           |        |        |        |        |      |
| 金额口径退款率（refund_rate_amt） |        |        |        |        |      |

写作要求：

- 如果销售额下降，要说明是订单数下降、客单价下降、售出件数下降，还是退款 / 折扣侵蚀。
- 如果销售额上升，要说明增长是否健康，例如毛利率是否同步改善、退款率是否上升。
- 如果没有对比周期，不要写“上升 / 下降”，只能描述当前水平和结构。

### 结构拆解

适用场景：

- 用户询问哪个商品、类目、品牌、渠道、门店、地区表现最好或最差。
- 用户询问销售变化来自哪里。
- 用户询问是否存在结构变化。

重点写：

- 哪些商品 / 类目 / 品牌 / 渠道 / 门店 / 地区贡献最大。
- 哪些维度拖累整体表现。
- 是否存在结构迁移。
- 结构变化是否能解释用户关心的问题。

推荐表格：

| 维度对象 | 销售净额 | 销售占比 | 订单数 | 售出件数 | 毛利率 | 变化率 | 说明 |
| -------- | -------: | -------: | -----: | -------: | -----: | -----: | ---- |
|          |          |          |        |          |        |        |      |

写作要求：

- 不要只列排名，必须解释排名和用户问题的关系。
- 如果 Top 商品销售高但毛利低，要提示“规模高但质量未必高”。
- 如果某类目贡献了主要下滑，要说明它对整体变化的贡献。
- 如果某渠道销售高但退款率也高，要提示销售质量风险。

### 原因分析

适用场景：

- 用户询问“为什么”。
- 用户询问变化原因。
- 用户询问某个业务问题背后的驱动因素。

按实际场景选择以下方向，不必全部写：

- 价格与折扣。
- 线上漏斗。
- 库存与可售性。
- 新客与复购。
- 促销与营销效率。
- 退款与退货。
- 渠道、门店、地区差异。

推荐表格：

| 可能原因   | 相关指标                     | 当前表现 | 对比表现 | 是否支持该原因 | 强度 |
| ---------- | ---------------------------- | -------: | -------: | -------------- | ---- |
| 订单量变化 | orders                       |          |          |                |      |
| 客单价变化 | aov_net                      |          |          |                |      |
| 折扣加深   | discount_rate                |          |          |                |      |
| 退款上升   | refund_rate_amt              |          |          |                |      |
| 库存限制   | stockout_rate / days_on_hand |          |          |                |      |

写作要求：

- 必须区分“已有证据支持的结论”和“目前只能作为线索的解释”。
- 不允许把描述性证据写成确定因果。
- 没有对照组、实验或充分排除混杂因素时，应使用“可能说明”“更可能”“线索表明”。
- 如果数据不足以解释原因，要明确说“当前数据只能定位现象，不能充分解释原因”。

## 关键证据

关键证据是对全文证据的压缩汇总，不替代核心发现部分的局部表格。

建议使用 Markdown 表格。

| 证据主题 |                                                         指标 | 当前值 | 对比值 | 分母/样本量 | 解释 | 强度 |
| -------- | -----------------------------------------------------------: | -----: | -----: | ----------: | ---- | ---- |
| 整体销售 |                                        销售净额（net_sales） |        |        |             |      |      |
| 订单规模 |                                             订单数（orders） |        |        |             |      |      |
| 客单变化 |                净额口径客单价（aov_net，AOV 指平均订单金额） |        |        |             |      |      |
| 商品结构 | 头部单品 / 类目销售占比（top sku/category share，SKU 指库存量单位/单品编码） |        |        |             |      |      |
| 毛利质量 |                                       毛利率（gross_margin） |        |        |             |      |      |
| 库存效率 | 库存周转次数 / 售罄率 / 库存可支撑天数（turnover / STR / DOH） |        |        |             |      |      |
| 客户质量 |         复购率 / 同期群表现（repeat_purchase_rate / cohort） |        |        |             |      |      |

要求：

- 关键证据不少于 4 条。
- 每条证据必须有分母、样本量或可比对象。
- 没有可比对象时不要硬写“上升 / 下降”。
- 表格中的证据必须能在核心发现中找到对应解释。
- 不要把正文没有解释过的新证据只放在这里。

## 业务解释

这一段回答“这些数字意味着什么”。

建议写法：

- 先写最重要的业务含义。
- 再写影响范围。
- 最后写持续观察点。

表达要求：

- 必须贴合用户问题，不要泛泛解释销售经营。
- 允许写“可能说明”。
- 不允许把描述性证据写成确定因果。
- 如果增长来自高折扣或高退款，必须明确提示“质量风险”。
- 如果销售下降同时伴随库存不足，要优先提示“不能直接判断需求下降”。
- 如果用户问的是具体动作建议，这一节应说明哪些结论能支持动作，哪些还只够支持继续验证。

---

## 限制与数据问题

必须单独成节，不允许埋在正文里。

至少检查并写出以下内容中的相关项：

- 缺失字段。
- 样本不足。
- 数据延迟。
- 成本口径不完整。
- 税费 / 运费 / 退款口径限制。
- 事件埋点不完整。
- 客户 ID 不稳定。
- 可比范围不足。
- 数据清洗对结果的影响。
- 是否只展示了部分表格，完整数据是否需要下载查看。

写法要求：

- 具体，不写空话。
- 写清限制影响的是哪条结论。
- 不要把“数据缺失”写成“业务上不存在该问题”。
- 如果核心发现中使用了局部样本或 Top N 表格，需要说明完整明细是否已另存。

---

## 结论与建议

建议拆成两部分。

### 结论摘要

- 用 2~4 句话总结最核心发现。
- 明确当前最可信的判断是什么。
- 如果存在多个可能解释，应区分“更可能的解释”和“仍需验证的解释”。
- 结论必须回扣用户原始问题。

### 后续建议

建议按优先级写：

- 立即动作。
- 短期验证。
- 后续补证据。

建议必须满足：

- 能从前文证据推出。
- 不要泛泛而谈。
- 如果证据只够提出“下一步验证动作”，就不要假装已经知道最终业务动作。
- 不要把经营建议写成绝对命令，应根据结论强度调整语气。

---

## 写作风格要求

- 全文使用 Markdown。
- 语气中性、直接、证据优先。
- 少写铺垫，多写和用户问题直接相关的发现。
- 少用形容词，多用数字和比较。
- 没有把握时，显式写“当前证据不足”。
- 不要伪精确，不要虚构原因。
- 如果有多种解释，应写“更可能 / 还需进一步验证”。
- 不使用强因果表达，除非上游证据明确支持因果判断。
- 不要把模板中的所有模块机械写满，应根据用户问题选择重点。
- 核心发现部分的篇幅必须明显多于问题与范围、限制说明等辅助部分。

---

## 默认篇幅建议

- 执行摘要：150~250 字。
- 问题与范围：80~150 字。
- 核心发现：400~700 字。
- 核心发现中的局部数据表格：1~3 个，每个 5~10 行。
- 关键证据表：4~8 条。
- 业务解释：150~300 字。
- 限制与数据问题：80~200 字。
- 结论与建议：150~300 字。

如果用户问题很具体，例如“哪个商品卖得最好”，可以进一步压缩执行摘要和问题范围，把篇幅集中在商品表现、排名表和解释上。

如果用户问题很复杂，例如“为什么销售额下降”，核心发现必须包含至少两个层次：

1. 整体变化拆解。
2. 关键驱动因素或拖累因素分析。

---

## 报告生成时的硬约束

- 不得新增上游未提供的事实。
- 不得把弱证据写成强结论。
- 不得省略限制说明。
- 不得把“数据缺失”隐去。
- 同一指标全篇保持同一口径。
- 若结论依赖估算字段，必须提示“估算”。
- 若上游分析轮次（analysis_rounds）中存在冲突证据，必须在报告中说明冲突，而不是只选择其中一方。
- 若上游只提供单期数据，不得写趋势判断。
- 若没有对比对象，不得写增长、下降、改善、恶化等比较性结论。
- 核心发现必须直接回应用户问题，不能只写通用经营概览。
- 核心发现中必须使用与用户问题相关的数据表格；如果数据量过大，只展示 Top N、Bottom N 或异常项，并提示完整数据文件。
- 关键证据表不能替代核心发现中的解释，正文必须解释关键数据为什么重要。
- 如果当前证据不足以回答用户最关心的问题，必须直接说明缺什么数据，而不是转移到其他无关分析。
