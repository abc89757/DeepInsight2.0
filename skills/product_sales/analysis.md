# 商品销售分析数据解释规则

## 总体原则

- 先判断数据能不能支持解释，再判断数据说明了什么。
- 先做描述性分析，再决定能否做归因性表达。
- 先报告证据，再报告推断。
- 先写确定性高的结论，再写方向性判断。
- 不把“没有看到证据”写成“证据证明没有问题”。

## 数据质量检查清单

每轮分析开始前，先检查以下问题：

### 完整性

- 关键字段是否存在：
  - 订单 ID（order_id）
  - 商品 ID（item_id）
  - 金额字段（amount）
  - 数量字段（qty）
  - 日期字段（date）
- 是否存在大量空值、未设置值（not set）、未知值（unknown）、游客客户（guest）。

### 一致性

- 订单头表和订单商品明细表的金额能否对齐。
- 退款表与订单表能否正确关联。
- 多张表中同一商品、门店、渠道的编码是否一致。

### 准确性

- 是否出现负销量、异常高单价、重复订单、重复事件。
- 库存是否出现不合理的负值。
- 成本是否异常为 0，或长期保持不变。

### 时效性

- 数据是否存在延迟。
- 当前统计周期是否尚未完整结束。
- 新近日期的数据是否存在回填问题。

### 可解释性

- 指标口径是否明确。
- 税费、运费、折扣、退款是否独立列示。
- 多币种、多时区是否已经统一。

### 可比性

- 对比的时间窗口是否一致。
- 对比的对象集合是否一致。
- 是否把促销期与非促销期直接进行硬性对比。

如果任一关键检查失败：

- 允许输出“数据问题”。
- 不允许输出强结论。

## 数据清洗与无用信息处理规则

数据清洗的目标不是“让数据看起来更好”，而是让后续分析基于更清晰、更一致、更可解释的数据。清洗过程必须保留处理痕迹，不能为了得到某个结论而选择性删除数据。

### 总体原则

- 先识别杂乱信息，再决定是否清洗。
- 能标准化的先标准化，不要直接删除。
- 能标记的先标记，不要静默丢弃。
- 只有确认无分析价值、无业务含义、且不会影响统计口径的数据，才允许排除。
- 清洗前后都应保留样本量，避免清洗导致结论失真。
- 如果清洗规则会显著影响结果，必须在分析结论或限制说明中写出。

### 常见杂乱信息类型

#### 1. 无关字段

例如：

- 页面展示字段
- 系统日志字段
- 临时导入字段
- 无业务含义的备注字段
- 重复存储但口径不明的派生字段

处理规则：

- 不参与指标计算。
- 不作为分析依据。
- 如果字段可能包含业务信息，例如备注中包含退款原因、促销说明，可以保留为辅助解释字段，但不能直接参与数值统计。

#### 2. 无效记录

例如：

- 测试订单
- 内部订单
- 已取消订单
- 未支付订单
- 空订单
- 金额和数量都为 0 的记录
- 明显由系统测试、爬虫、压测产生的记录

处理规则：

- 默认从正式销售指标中排除。
- 如果用户明确要求分析取消订单、测试订单或异常订单，则可以单独保留为异常分析对象。
- 排除前应确认这些记录有明确标识，不要仅凭金额异常就直接删除。

#### 3. 重复记录

例如：

- 同一订单重复导入。
- 同一事件重复上报。
- 同一退款记录重复写入。
- 订单头表与订单明细表 join 后导致行数膨胀。

处理规则：

- 订单类指标按订单 ID 去重。
- 客户类指标按客户 ID 去重。
- 事件类指标优先根据事件 ID 去重。
- 如果无法确认是否重复，应标记为“疑似重复”，不要直接删除。
- 如果 join 后数量异常放大，应优先检查连接键，而不是继续分析结果。

#### 4. 空值、未知值和占位值

例如：

- null
- 空字符串
- unknown
- not set
- N/A
- -
- 其他
- 未知
- guest

处理规则：

- 关键字段缺失时，不用于对应指标计算。
- 维度字段缺失时，可以归为“未知”组，但不能忽略其占比。
- 如果“未知”占比很高，应作为数据质量问题上报。
- 不要把 unknown、not set、guest 直接解释成真实业务类别。

#### 5. 文本字段脏数据

例如：

- 前后空格。
- 大小写不统一。
- 全角半角不统一。
- 同一渠道有多个写法，如 “抖音”、“douyin”、“Douyin”。
- 同一地区有多个写法，如 “广西”、“广西省”、“广西壮族自治区”。

处理规则：

- 可进行去空格、大小写统一、全角半角统一。
- 可将明显同义的类别做标准化映射。
- 标准化映射必须保留规则说明。
- 不确定是否同义时，不要擅自合并。

#### 6. 异常数值

例如：

- 负销量。
- 异常高单价。
- 异常大订单。
- 成本为 0。
- 折扣率超过 100%。
- 库存为负数。
- 转化率大于 100%。

处理规则：

- 优先标记异常，而不是直接删除。
- 如果异常值明显来自录入错误或系统错误，可以从主分析中排除，但应记录排除数量。
- 如果异常值可能是真实业务现象，例如大客户批量采购，应保留，并单独说明。
- 对均值类指标影响很大的异常值，应同时考虑中位数、分位数或截尾均值作为辅助观察。

#### 7. 时间字段异常

例如：

- 订单时间晚于退款时间。
- 支付时间早于下单时间。
- 发货时间早于支付时间。
- 当前周期数据尚未完整。
- 新近日期存在数据回填延迟。

处理规则：

- 时间逻辑不合理的记录应标记为异常。
- 当前周期未结束时，不要直接与完整历史周期硬比。
- 存在回填延迟时，应降低结论强度。
- 同一指标必须使用同一种时间口径。

#### 8. 无业务意义的噪声类别

例如：

- 类目字段中出现“其他”占比过高。
- 渠道字段中出现大量未知来源。
- 商品名称中包含乱码。
- 地区字段粒度混乱。
- 促销字段中包含无法识别的活动名。

处理规则：

- 噪声类别占比较小时，可以归入“其他 / 未知”。
- 噪声类别占比较高时，必须作为数据质量问题上报。
- 不要对高占比噪声类别做强业务解释。
- 如果噪声类别影响主要结论，应要求补充更清晰的数据字段。

### 清洗动作分级

你对杂乱数据的处理应分为四类：

1. 保留
   - 数据虽然不完美，但仍有明确业务含义。
   - 可以参与分析，但需要说明限制。

2. 标准化
   - 数据含义明确，只是格式不一致。
   - 例如大小写统一、去空格、渠道名称合并。

3. 标记
   - 数据可能异常，但不能确认错误。
   - 例如异常高单价、大额订单、疑似重复记录。

4. 排除
   - 数据明确无效，且会污染正式指标。
   - 例如测试订单、未支付订单、重复导入记录。

### 清洗后的解释规则

- 如果清洗前后核心指标变化很小，可以正常分析。
- 如果清洗后核心指标明显变化，必须说明清洗对结果的影响。
- 如果大量数据被排除，结论强度应下调。
- 如果无用信息或脏数据占比过高，应优先输出“数据质量问题”，而不是强行业务分析。
- 如果清洗规则不确定，应保留多个口径结果，例如“含异常值口径”和“剔除异常值口径”。

## 默认解释顺序

### 先解释销售结果，再解释原因

如果销售净额（net_sales）发生变化，默认按下面顺序拆解：

1. 订单数（orders）是否变化。
2. 净额口径客单价（aov_net，AOV 指 Average Order Value，平均订单金额）是否变化。
3. 净额口径平均售价（asp_net，ASP 指 Average Selling Price，平均销售单价）是否变化。
4. 折扣率（discount_rate）是否变化。
5. 商品结构是否变化。
6. 渠道 / 门店 / 地区结构是否变化。
7. 库存是否限制了可售数量。
8. 新客 / 老客 / 复购是否变化。

### 收入变化的默认拆法

- 销售额变化，优先拆成：
  - 订单量变化。
  - 客单价变化。

- 客单价变化，优先拆成：
  - 每单件数变化。
  - 平均售价变化。
  - 折扣强度变化。

- 毛利变化，优先拆成：
  - 销售净额变化。
  - 成本变化。
  - 商品结构变化。
  - 促销变化。

## 结构性差异与真实价格差异的区分

当用户询问“为什么 A 比 B 贵”或“为什么某渠道客单价更高”时，不要直接把整体均值差异解释成真实价格差异。

默认检查顺序：

1. 商品结构是否一致
   - 类目、品牌、规格、套装比例是否不同。

2. 客户结构是否一致
   - 新客 / 老客、地区、会员等级、企业客户比例是否不同。

3. 渠道结构是否一致
   - 自营、平台、门店、直播、分销等渠道结构是否不同。

4. 折扣与促销是否一致
   - 是否某一组更常参加折扣活动或促销活动。

5. 在匹配单品（SKU，Stock Keeping Unit，库存量单位/单品编码）、类目、规格、渠道后，再比较售价
   - 只有匹配后差异依然稳定，才允许解释为“真实价格差异”。

## 样本量不足时的降级规则

以下为项目默认阈值，可根据业务规模调整。

### 低样本

满足任一条件时，标记为“弱证据”：

- 订单数 < 30。
- 购买客户数 < 20。
- 商品售出件数 < 30。
- 线上浏览数 < 100。
- 关键分母缺失。

输出方式：

- 只描述现象。
- 不给强解释。
- 明确写出“样本不足”。

### 中样本

满足以下条件之一，可标记为“中等证据”：

- 30 <= 订单数 < 100。
- 100 <= 浏览数 < 1000。

输出方式：

- 可以给方向性解释。
- 必须带分母和限制说明。

### 高样本

满足以下条件之一，可标记为“较强证据”：

- 订单数 >= 100。
- 浏览数 >= 1000。
- 同一模式在多个维度重复出现。

输出方式：

- 可以给较强描述性结论。
- 如果没有对照组，仍不要使用强因果措辞。

## 结论强度标签

每条结论都建议附带强度标签：

### 强

适用条件：

- 样本充足。
- 至少两类指标相互印证。
- 关键替代解释已排查。
- 指标口径清晰。

### 中

适用条件：

- 主模式清楚。
- 样本基本足够。
- 仍存在部分未排查因素。

### 弱

适用条件：

- 只来自单一指标。
- 分母过小。
- 数据延迟或口径不清。
- 只能视为分析线索。

## 常见模式解释库

### 线上漏斗

- 浏览稳定、加购下降
  - 优先怀疑价格吸引力、详情页表达、商品匹配度、竞争替代等问题。

- 加购稳定、结账下降
  - 优先怀疑购物车门槛、优惠券失效、配送费用、库存可售性等问题。

- 结账稳定、购买下降
  - 优先怀疑支付环节、地址 / 运费问题、库存锁定失败、履约承诺下降等问题。

- 购买增长、退款同步增长
  - 说明销售增长质量可能不足，优先检查商品质量、商品描述偏差、履约体验等问题。

### 商品经营

- 销售额增长但毛利率下降
  - 多见于高折扣、低毛利商品占比提高、成本上升等情况。

- 销量增长但销售额不增长
  - 多见于降价、规格下沉、低价单品（SKU，Stock Keeping Unit，库存量单位/单品编码）占比提高等情况。

- 高销量但低毛利
  - 说明该商品可能是“冲量品”，不应直接当成最优商品。

- 低销量但高毛利
  - 可能是高价值长尾商品，不应直接清退。

### 库存经营

- 售罄率 / 动销消化率高，库存可支撑天数低，缺货较多
  - 对应指标：
    - 售罄率 / 动销消化率（sell_through_rate）
    - 库存可支撑天数（DOH，Days on Hand）
    - 缺货天数 / 缺货率（stockout_days / stockout_rate）
  - 解释：商品热销，但补货不足。

- 库存周转次数低，库存可支撑天数高，销量弱
  - 对应指标：
    - 库存周转次数（inventory_turnover）
    - 库存可支撑天数（DOH，Days on Hand）
  - 解释：压货风险较高。

- 销量下降但库存也很低
  - 不能直接判断需求转弱，应先排除“没货卖”的可能。

### 客户经营

- 新客增长、复购下降
  - 可能说明拉新质量下降，或老客经营变弱。

- 老客贡献高、新客不足
  - 说明当前销售可能较稳定，但中期增长风险上升。

- 同期群首购表现强、后续留存弱
  - 同期群（cohort，指按首购时间等共同特征划分的一组客户）后续表现较弱，可能说明首单主要由促销驱动，不一定代表高质量增长。

## 因果表达规则

只有满足以下条件之一，才允许使用“导致 / 带来 / 提升 / 下降由……造成”等偏强因果措辞：

- 有随机实验。
- 有 A/B 测试（A/B test，指把用户或对象分成实验组和对照组，用于比较不同策略效果的实验方法）。
- 有明确自然实验。
- 有可靠对照组。
- 有同口径前后对比，且主要混杂因素已排除。

否则统一使用以下表达：

- 相关。
- 同步变化。
- 伴随变化。
- 可能解释。
- 线索表明。

## 分析轮次记录（analysis_rounds）建议沉淀字段

每一轮分析建议沉淀以下字段：

- 本轮分析目标（round_goal）
- 分析范围（scope）
- 使用的指标（metrics_used）
- 证据项（evidence_items）
- 分组 / 分层发现（segment_findings）
- 关键驱动因素（key_drivers）
- 异常现象（anomalies）
- 数据质量问题（data_quality_issues）
- 置信水平 / 结论强度（confidence_level）
- 未解决问题（unresolved_questions）
- 下一步需要补充的证据（next_evidence_needed）
- 暂定结论（provisional_conclusion）

## 数据处理节点（DataProcessorNode）的额外职责

- 对 SQL 输出结果做二次计算。
- 对异常值做标记，而不是静默删除。
- 输出维度分母，避免只有比率没有样本量。
- 发现数据问题时优先上报，不强行补故事。
- 如果结果相互冲突，显式记录冲突，而不是私自选择其中一方。
