有，最开始聊的东西比这次 commit 覆盖得更大。现在主要剩这些：

1. **文件数据源先搁置了**
   之后要支持用户上传数据文件，不只是数据库。到时候要决定是：
   - 把 `SQLEngineerNode` 改名成更通用的 `DataExtractorNode`
   - 还是保留 SQL 节点，再新增一个文件取数 Agent，由 `DataProcessorNode` 决定找谁取数

2. **Schema 节点后面要扩展**
   现在数据库 schema 是在 `LoadSchemaNode` 获取的。之后加文件时，它可能要变成更通用的“数据源概览节点”，既能读数据库 schema，也能读上传文件的字段、样例、行数、类型分布等概览。

3. **SQL 节点命名和职责还没最终定**
   当前代码里还是 `SQLEngineerNode`。如果未来它只负责数据库 SQL，可以保留；如果它要抽象成“取数工程师”，就要改名、改 prompt、改状态字段命名。

4. **`sql_rule` 的处理还可以收尾**
   我们已经决定 SQL 规则不作为 skill，而是写进 SQL 节点 system prompt。但你那个 `skills/product_sales/sql_rules.md` 现在应该还是未跟踪文件。后面要么把里面真正有价值的规则并入 prompt，要么删掉/不提交。

5. **DataProcessor 的决策方式还可以观察**
   现在是用工具参数决定“需要数据 / 开始分析”。这个方案能跑，但还需要实际多测几轮，看模型会不会稳定地最后调用决策工具。效果不好时，可以换成结构化输出或“文本 + 末尾固定 JSON”。

6. **报告流式已经接上了，但可以再跑一次端到端验证**
   这次前端已经接入报告窗口流式输出了，不过最好用真实任务跑一遍，看报告窗口是否从 `report_writer` 开始就逐字出现，最终报告是否还能正常下载 PDF。

7. **节点说明文档还没正式整理**
   一开始你问过“各个 Node 接受哪些信息、读哪些 skill、Node 内是否保存记忆”。我们聊过，但还没把最新版整理成一份稳定文档。这个很值得做，因为现在 graph 职责变清楚了。

8. 对于调用工具读取数据这一点
    这一轮生成完后，把记忆清一清，不要让他记住这么多数据

9. 要增加能让Agent立即生成结果的选项
    这点也许要在stream_agent方法里提前埋好特定的事件，如果检测用户点击了就触发某个操作，不过我现在没什么很好的想法

10. 定下分析方向的时候，应该把裁判判断为什么不收敛的原因也加回去。也就是不要让裁判只输出是否收敛了，而是输出原因，至于是否收敛的标识就通过工具调用吧

11. 关于自沉淀skill，应该在分析任务选了通用skill后给出原因为什么现有的skill不能满足该任务，以及应该需要一个什么样的skill
